# System Design Write-Up
## 1. Compatibility Scoring Design
Each tenant-listing pair is scored once and persisted to the CompatibilityScore collection, rather than recomputed on every request — a compound unique index on (tenant, listing) enforces this. A cached score is considered valid unless invalidated by one of two conditions. First, a 7-day age limit. Second — and more important — a score is invalidated if TenantProfile.updatedAt or Listing.updatedAt is newer than the score's computedAt. This second condition was added after manual testing surfaced a real bug: a tenant who edited their budget or preferred city mid-session kept seeing their old score, since the original design only checked age. Comparing update timestamps against computedAt makes the score responsive to actual user behavior instead of just elapsed time.
Scores are computed synchronously when a tenant requests a page of listings, rather than via a background batch job across the whole database. This adds latency the first time an unseen listing is scored, but avoids an O(N×M) explosion of LLM calls by only scoring pairs a tenant is actively viewing.

## 2. LLM Integration and Fallback
Scoring is split into two independent services behind one orchestrator, getOrComputeScore(). The LLM path calls Groq's chat completions endpoint (llama-3.3-70b-versatile), sending tenant location/budget/move-in date/preferences alongside listing location/rent/availability/description, instructed to return strict JSON with no markdown. Because Llama models sometimes wrap JSON in code fences despite instructions, the parser strips fences defensively before JSON.parse, then validates score is a number in [0,100] and explanation is non-empty. An 8-second AbortController timeout guards against hangs. Any failure — malformed JSON, an out-of-range score, timeout, or API error — throws a typed LLMParseError rather than returning bad data silently.
The orchestrator catches that error unconditionally and falls back to a deterministic rule-based scorer: start at 100, subtract a penalty proportional to how far rent sits outside the tenant's budget, subtract a fixed penalty for a city mismatch, a smaller one for an area mismatch within the same city. Every score — either path — is tagged method: "llm" | "rule-based" before persisting, which let us independently verify the fallback: deliberately invalidating the Groq API key produced a console.warn, a rule-based score, and a normal 200 response, not a 500 — proving the fallback is genuinely non-blocking, not just present in the code.

## 3. Chat Implementation
Real-time messaging uses Socket.io rooms scoped to accepted interests (interest:{interestId}), so a channel only meaningfully exists once both parties have agreed to talk. Authorization is enforced twice, independently — once on join_room, again on every send_message — since an interest's status can change between those two events (e.g. an owner declining mid-conversation), and trusting a stale "already joined" state would be a real gap.
Production deployment (client on Vercel, server on Render, different origins) surfaced a subtler problem. Our REST API uses an httpOnly JWT cookie, proxied through Vercel rewrites to stay first-party — but Socket.io must connect directly to Render, since Vercel's serverless functions can't hold long-lived WebSocket connections. That genuinely cross-origin socket connection triggered Safari's ITP, which silently stripped the cookie from the handshake, breaking chat only in Safari, only in production. The fix issues a short-lived (15-minute), purpose-scoped secondary JWT alongside the existing /api/auth/me response, held only in React state (never persisted, avoiding localStorage XSS exposure), and passed to Socket.io via a synchronous auth callback function rather than an async pre-connect fetch — avoiding both the ITP cookie block and a React unmount race condition an earlier async-token approach had caused.

## 4. Notification Flow
Two notification channels run in parallel and independently: email (Nodemailer over Gmail SMTP) and in-app Notification documents. Email fires conditionally — the owner is notified only when a tenant's compatibility score exceeds 80, matching the spec's high-compatibility threshold — while accept/decline outcomes always email the tenant. In-app notifications, by contrast, are created unconditionally for both event types, since they don't depend on an external service that can fail.
Every email send is wrapped in a try/catch that logs failure but never blocks or rolls back the underlying interest action. This was verified directly: sending to a real inbox confirmed delivery and correct HTML rendering (including the dynamic compatibility score), while sending to a deliberately invalid domain confirmed a clean bounce with zero impact on the API response. The in-app notification remains the source of truth, guaranteeing the user sees the update regardless of email deliverability.
